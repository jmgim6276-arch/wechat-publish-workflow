# 公众号工作流接入方案

## 目标
将 `wechat-publish-workflow-skill` 作为 AI 工作流入口，将 `toolkit/` 作为真实执行层，形成“素材采集 → 成稿 → 样式调整 → 草稿发布/更新 → 回拉同步”的完整链路。

## 目录结构

```text
gongzhhao.project/
├── skills/
│   └── wechat-publish-workflow-skill/
│       ├── SKILL.md
│       ├── references/
│       └── agents/
├── toolkit/
│   ├── publish_article.py
│   ├── update_article.py
│   ├── fetch_draft.py
│   ├── wechat_tech_converter.py
│   ├── make_cover.py
│   ├── config.yaml.example
│   └── requirements.txt
└── docs/
    └── 公众号工作流接入方案.md
```

## 职责分层

### 1. skill 层：`skills/wechat-publish-workflow-skill`
负责：
- 触发词识别（如“我要开始写公众号了”）
- 对话素材采集与孵化
- 从经历/情绪/判断转化为文章结构
- API 健康检查、白名单、凭据排查
- 决策“新建草稿”还是“覆盖更新”
- 风格迭代规则（封面与正文分离）

### 2. 执行层：`toolkit/`
负责：
- `publish_article.py`：新建草稿
- `update_article.py`：覆盖更新已有草稿
- `fetch_draft.py`：回拉远端草稿、对比修改
- `wechat_tech_converter.py`：Markdown → 微信正文 HTML
- `make_cover.py`：封面生成
- `config.yaml.example`：公众号配置模板

## 工作流映射

### A. 开始写公众号 / 先攒素材
走 skill：
- 由 `SKILL.md` + `references/` 指导采集和孵化
- 不直接触发发布脚本

### B. 素材成熟，准备发草稿
走 skill 决策 + toolkit 执行：
- 新稿：`publish_article.py`
- 旧稿覆盖：`update_article.py`

### C. 正文样式不对
优先调整：
- `wechat_tech_converter.py`

### D. 封面不对味
优先调整：
- `make_cover.py`

### E. 戴总改过草稿后，需要同步回本地
执行：
- `fetch_draft.py`

## 配置建议

1. 复制配置模板：

```bash
cp toolkit/config.yaml.example toolkit/config.yaml
```

2. 填写公众号参数：
- `appid`
- `secret`
- `author`
- `mp_name`

3. 安装依赖：

```bash
python3 -m venv ~/.wechat-venv
~/.wechat-venv/bin/pip install -r toolkit/requirements.txt
```

## 建议使用顺序

1. 先用 skill 采集和整理内容
2. 用 converter 调整正文呈现
3. 用 cover 工具调整封面
4. 用 publish/update 推送草稿
5. 用 fetch 回拉戴总修改

## 后续可扩展方向

- 增加 `media_ids.json` 统一记录机制
- 增加 `create vs update` 的自动决策 glue 脚本
- 把 skill 中提到的 token/白名单检查脚本沉淀到 `toolkit/diagnostics/`
- 增加 `.env.example` 或统一配置层，减少多配置源混乱

## 当前结论

最优方案不是二选一，而是：
- `wechat-publish-workflow-skill` 做“脑子”
- `toolkit/` 做“手脚”

也就是：
**skill 管流程，toolkit 管执行。**
